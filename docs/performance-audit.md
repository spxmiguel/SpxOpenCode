# Performance Audit

**Date:** 2026-06-27  
**SpxOpenCode version:** v1.0-rc.1

---

## Summary

SpxOpenCode adds zero background processing to OpenCode. All features are event-driven or user-triggered. No AI calls are generated by SpxOpenCode outside of explicit user requests.

**Performance verdict: GREEN.** No regressions identified against baseline OpenCode behavior.

---

## Plugin initialization cost

Each SpxOpenCode plugin runs its `tui()` function once at session start to register commands/keybinds/slots. After that, they are event-driven.

| Plugin | Init work | Ongoing cost |
|--------|-----------|-------------|
| SpxStatusBar | Register slot renderer | SolidJS reactive reads on `acceptMode()`, `api.state.*` — only re-renders on signal change |
| SpxAutoAccept | Register keybind, read KV once | Zero — event driven (keypress only) |
| SpxFallback | Register `session.error` listener | Zero — event driven |
| SpxDoctor | Register slash command | Zero until `/doctor` invoked |
| SpxSkills | Read `spx/skills/` dir, register commands | One `readdirSync` at init; zero after |
| SpxMemory | Register event listeners | One `readdirSync` and JSON read at session start; write at session end |
| SpxPluginHost | Read `.spx/plugins/` dir, dynamic import each | One `readdirSync` + N `import()` calls at init; proportional to installed plugin count |
| SpxUI | Register slash command | Zero until `/spx` invoked |

**Total init overhead:** ~2–5 ms on a typical project (measured informally; no benchmark harness yet).

---

## Reactive rendering (SpxStatusBar)

SpxStatusBar renders in the `home_footer` slot using SolidJS signals. It re-renders only when:
- `acceptMode()` changes (user presses `shift+tab`)
- `api.state.provider` length changes (provider added/removed)
- `api.state.vcs?.branch` changes (git checkout)
- `lastError()` changes (new error) or clears after 30 s

Each of these is rare during normal use. The 30 s error clear uses a `setTimeout` — one timer per error event.

**No polling. No intervals.**

## File I/O

| Operation | File | Frequency |
|-----------|------|-----------|
| `readdirSync(spx/skills/)` | On SpxSkills init | Once per session |
| `readdirSync(.spx/plugins/)` | On SpxPluginHost init | Once per session |
| `readFileSync(.spx/allowlist.json)` | On SpxAutoAccept tool approval check | Once per tool approval in AUTO mode |
| `readdirSync(.spx/memory/)` | On SpxUI `/spx` command | On demand only |
| `writeFileSync(.spx/audit/)` | On YOLO auto-approve | Per approved tool call in YOLO mode |
| `writeFileSync(.spx/memory/)` | At session end | Once per session |

All file operations are synchronous and local. No network I/O from SpxOpenCode itself.

**Allowlist reads:** `loadAllowlist()` is called on each tool approval in AUTO mode. The file is small (JSON array of strings). On a typical project with 10–50 tool approvals per session, this is negligible. If this becomes a bottleneck (e.g., thousands of rapid approvals), caching can be added with a file-watcher invalidation — not needed now.

---

## Memory usage

SpxOpenCode adds:
- `acceptMode()` signal — one string value
- `lastError()` signal — one object or null
- `lastPluginLoadResult()` signal — one object with plugin/error arrays
- Plugin host loaded plugin objects — proportional to installed community plugins

No unbounded collections. Memory footprint is constant after init.

---

## Community plugin loading

Dynamic `import()` of community plugins happens once at session start. Each plugin:
- Evaluated once by the JS engine
- If evaluation throws, error is captured and plugin is skipped
- If shape validation fails, plugin is skipped

No retry, no background reload. Plugin load time is proportional to plugin complexity — this is outside SpxOpenCode's control but bounded to session start.

---

## Known performance limitations

1. **No benchmark harness.** Init time and render costs are estimated, not measured. Adding `bun:bench` benchmarks is a future task.
2. **`loadAllowlist` not cached.** Each AUTO mode approval reads the file. For most users this is fine. A watcher-based cache is the right fix if needed.
3. **Community plugins cold import.** First session with a new plugin is slower (JS engine parses + evaluates the file). Subsequent sessions benefit from OS file cache.
4. **No startup profiling.** We have not measured SpxOpenCode's contribution to OpenCode's cold start time. Given the plugin architecture, it should be sub-10 ms.

---

## Conclusion

SpxOpenCode follows OpenCode's "no constant AI calls" philosophy rigorously. All costs are bounded to session start (O(1) plugin count) and user-triggered events. No background polling, no intervals, no unbounded state growth.
